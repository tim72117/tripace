# 整趟行程的建立順序

## 核心規則

**區間事件先建，單點事件後建。**

歸組邏輯是「時間重疊就歸入現有行程」。如果先建單點事件（景點/餐廳），它會自己開一個新 Trip；後面再建住宿時，反而被歸進那個錯誤的 Trip。正確作法是讓住宿先建立 Trip 的時間骨架，之後所有在這段期間的事件才能正確歸入。

---

## 建立順序

```
1. 機票（去程）          ← 確立整趟行程的起訖時間
2. 住宿                  ← 撐開每天的時間骨架
3. 機票（回程）          ← 同上
4. 景點 / 餐廳 / 活動   ← 時間落在骨架內，自動歸入正確行程
```

### 為什麼這個順序

| 事件類型 | 特性 | 說明 |
|---|---|---|
| 機票 | start → end，跨多日 | 最大的時間區間，定義行程邊界 |
| 住宿 | start → end，通常多晚 | 填滿中間每天的時間骨架 |
| 景點 / 餐廳 | 單一時間點或半天 | 時間點落在住宿區間內，自動歸入同一個 Trip |

---

## 實際範例：2025 宮古島

### 錯誤順序（先建景點）

```
record "浮潛 6/20"
  → candidates 空，開了新 Trip「浮潛」(6/20)

record "宿希爾頓 6/19~6/21"
  → candidates 有「浮潛」Trip，時間重疊
  → 歸入「浮潛」Trip ← 錯誤，行程名稱變成景點
```

### 正確順序（先建住宿）

```
record "宿希爾頓 6/19~6/21"
  → candidates 空，開了新 Trip「宿希爾頓」(6/19~6/21)

add_to_trip(tripID="", title="2025 宮古島")
  → 把 Trip 重新命名（或建立後更名）

record "浮潛 6/20"
  → candidates 有「2025 宮古島」，6/20 落在 6/19~6/21 內
  → 歸入「2025 宮古島」← 正確
```

## 完整行程的建立示範

假設這是一趟「2026 北歐芬蘭（7/3~7/4）」行程，使用者一次輸入：

```
桃園出發 7/3 23:30
赫爾辛基轉機 7/4 14:00
羅瓦涅米聖誕老人村 7/4 16:00
宿 Arctic TreeHouse Hotel 7/4~7/5
回程 7/5 06:00
```

**LLM 的處理順序：**

```
第一批：區間事件（建立骨架）
  1. record "出發 桃園→赫爾辛基", start=7/3 23:30, end=7/4 14:00
     → candidates 空，開了新 Trip（7/3~7/4）

  2. record "宿 Arctic TreeHouse Hotel", start=7/4, end=7/5
     → candidates 有上面那個 Trip，時間重疊
     → 歸入同一 Trip，Trip 時間擴張到 7/3~7/5

  3. record "回程", start=7/5 06:00
     → candidates 有同一 Trip，歸入

第二批：單點事件（填入骨架）
  4. record "赫爾辛基轉機", start=7/4 14:00
     → candidates 有 Trip，7/4 在 7/3~7/5 內
     → 歸入

  5. record "聖誕老人村", start=7/4 16:00
     → 同上，歸入
```

結果：所有 5 筆條目都在同一個 Trip 裡，行程完整。

---

## 多段行程同時輸入的處理

若使用者一次輸入多趟不同行程（如「7月芬蘭 + 8月冰島」），每趟的骨架事件要在對應趟次的單點事件前建立：

```
✅ 正確：
  芬蘭機票 → 芬蘭住宿 → 芬蘭景點
  冰島機票 → 冰島住宿 → 冰島景點

❌ 錯誤：
  芬蘭景點 → 芬蘭住宿（骨架太晚，景點已自立 Trip）
```

---

## 沒有時間的條目

沒有填 `start` 的條目不參與歸組，不會建立也不會歸入任何 Trip。記錄完後直接結束，不呼叫 `add_to_trip`。

---

## 小結

| 原則 | 說明 |
|---|---|
| 先區間，後單點 | 住宿 / 機票先建，景點後建 |
| 名稱在第一筆決定 | Trip 名稱預設為第一筆 entry 的 item，之後再 update_trip 改名 |
| 時間骨架撐得愈大愈好 | 骨架時間範圍要完整涵蓋所有單點事件的日期 |
| 跨趟不混 | 不同行程的時間不重疊，系統自然拆成不同 Trip |
