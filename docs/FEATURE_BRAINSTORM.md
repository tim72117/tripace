# 功能腦力激盪:旅遊編排軟體可以做到什麼

> 產生方式:2026-07-20 由三個獨立視角的腦力激盪 agent 平行發想(每視角 10 項),再交集統整為 10 個項目。
> 三個視角:**繁瑣痛點簡化**(旅行全程有哪些煩人的工作可以被消除)、**AI 獨特能力**(傳統表單+拖曳做不到、LLM+工具才做得到的事)、**使用者吸引力**(第一分鐘的魔法時刻/過程黏著/長期與傳播)。
> 統整原則:三視角或雙視角同時命中的項目優先(獨立視角收斂到同一點是最強的需求訊號),其餘擇優;落選但值得記錄的項目收在附錄。

## 統整後的 10 個項目

### 1. 靈感收件匣(貼上即成行程)⭐ 三視角命中
把 IG/小紅書/YouTube/部落格連結、截圖、朋友 LINE 訊息、甚至研討會議程整段貼進來,AI 抽出裡面的地點/場次、比對成 Google Places 真實地標,收進候選池並建議可插入的日子時段。

- **取代的繁瑣流程**:相簿裡幾十張美食截圖、瀏覽器一堆書籤,排程時一張張回看、手動搜地名、複製到備忘錄再抄進行程。
- **為什麼吸引人**:一般 AI 行程產生器憑空生出罐頭行程;把「你自己收藏的東西」變成行程,第一分鐘就是「這是我的行程」而不是「AI 的行程」。
- **AI 角色**:核心——從雜亂自由文本/截圖抽實體、消歧義(哪家分店)、判斷塞哪裡合理。多模態 LLM + Places Text Search。
- **可行性**:LLM 抽取 + Places 比對(已接)可跑通;IG/小紅書反爬,走截圖路線比抓連結內文穩。同名地點解析錯誤會把人導去錯的店,結果必須讓使用者確認。
- **延伸**:研討會場景(貼議程 → 個人課表 + 衝堂偵測)用同一套機制。

### 2. 行程健檢(合理性檢查)⭐ 三視角命中
自動掃描整份行程,標紅「這天週一休館」「兩點之間車程 90 分鐘塞不下」「末班車 22:10」「這天走 12 公里」「六小時沒排吃飯」,每項附一鍵修正建議。

- **取代的繁瑣流程**:一個個景點開 Google Maps 查營業時間對照星期幾、自己一段段查交通心算出發時間;週一排了三個休館的博物館,到現場才發現。
- **為什麼吸引人**:AI 行程產生器最常被罵的就是排出地理亂跳、踩到休館日的行程;第一次看到 app 主動抓出「你這天根本走不完」,信任感立刻建立。
- **AI 角色**:單項檢查是規則(不需 AI);「發現問題 → 結合情境解釋嚴重性 → 提出具體修法」的閉環需要 LLM。
- **可行性**:Places Details 營業時間現成已接(速贏);交通需 Routes/Distance Matrix API(有成本,需快取)。誤報率是命門——狼來了幾次使用者就永久關掉。

### 3. 對話式重排(模糊指令 + 連鎖順延 + diff 預覽)◆ 雙視角命中
一句「第二天下午太趕,幫我鬆一點」讓 AI 重排整段;刪掉或移動一個點時自動把後續順延補位。所有批次改動先以 diff 預覽(移了什麼、砍了什麼、為什麼),確認後一鍵套用,並可一鍵還原。

- **取代的繁瑣流程**:臨時取消一個點後,手動逐一改後面五個項目的時間、重查每段交通,改到一半放棄導致整天亂掉。
- **為什麼非 AI 不可**:「太趕」「鬆一點」是主觀約束,傳統 UI 沒有這種輸入欄位;LLM 把模糊意圖翻譯成一組具體編輯操作,人只負責審核。
- **可行性**:既有 LLM 工具呼叫管線直接可用;關鍵新增是 diff 預覽 + undo——沒有它,AI 一次改動過大就會信任崩潰。這也是後續所有 AI 批次操作的信任基礎建設。

### 4. 接駁自動補齊(交通是一等公民)
景點「之間」自動補上:搭哪班車、多久、約多少錢,依情境選交通方式(拖行李 → 計程車、有 JR Pass → 優先 JR),在斷點(末班車趕不上、假日停駛)主動警示。

- **取代的繁瑣流程**:自己開地圖一段段查 A→B 幾分鐘、有沒有末班車,行程一改全部重查。
- **為什麼非 AI 不可**:導航 App 只回答「現在從 A 到 B」,不懂你的脈絡;規劃期的交通推理(考慮行李、票券、時段)是 AI 的事。
- **可行性**:Google Routes/Directions API(含大眾運輸班次)直接可做;難點是呼叫成本(需快取、只算相鄰段)與海外運輸資料品質。這也是「地理理解」呈現(時間軸相鄰點的距離與移動時間)的資料基礎。

### 5. 空檔雷達(時間感知推薦)
偵測尷尬空檔(「博物館 3 點結束、晚餐訂 6 點半」),依「那個時刻你人會在哪、還開著、體力剩多少、合不合口味」推薦補位選項。

- **與現有推薦的差異**:現有 recommend_nearby 是「以瀏覽位置為中心的靜態搜尋」;這是以**未來時刻的位置 + 當時營業狀態 + 前後行程脈絡**推理的多約束推薦,傳統篩選器組不出來。
- **可行性**:既有 Places Nearby/Text Search + 營業時間 + 時間軸脈絡,是現有推薦功能的直接升級。命中率依賴對體力/意願的猜測,需以低打擾建議卡呈現而非自動插入。

### 6. 旅伴共決(候選池 + 投票 + AI 折衷)◆ 雙視角命中
旅伴把想去的店丟進共享候選池,滑卡片表態(想去/不想/都可);每人也可私下跟 AI 說雷點,AI 彙整衝突(A 要逛街、B 要爬山)提出折衷方案(分頭兩小時、輪流主場日)給全員表決,高票項目由 AI 塞進合適時段。

- **取代的繁瑣流程**:「LINE 群組丟連結 → 沒人整理 → 回頭翻不到」;群組裡不好意思說「我不想去」的社交成本。
- **為什麼吸引人**:團體的決策發生在哪裡,行程就留在哪裡。Wanderlog 只有留言沒有表決。
- **可行性**:多人成員與權限機制已有,補候選清單資料表與即時同步(WebSocket 已有);AI 折衷是對話框架的延伸。候選池與第 1 項共用。

### 7. 一鍵複製 + AI 改編
看到別人公開分享的行程按「複製成我的」,選好日期人數後對 AI 說「我們只有四天、帶兩個小孩」,自動改編成自己的版本。

- **為什麼吸引人**:複製完只能手動慢慢改(Wanderlog 現況)vs「別人的五天行程一句話變成我的四天親子版」——新使用者第一分鐘拿到八成完成度;每個分享出去的行程變成會繁殖的資產,形成獲客迴圈。
- **可行性**:公開分享機制已存在,AI 改編就是現有對話規劃的一個預設 prompt,幾乎是白拿的功能。

### 8. 今天模式 + 現場救援 ◆ 雙視角命中
旅行當天打開 app 直接是「現在 → 下一站」大卡片,附導航入口與離線快取;途中一句「現在下大雨」「這間臨時公休」「我們晚了 40 分鐘」,AI 以目前位置與時間改寫今天剩餘行程,保住訂位等鎖定項目。

- **為什麼非 AI 不可**:站在路邊淋雨用手機拖曳排程是災難;「一句話 + GPS」重排是傳統 UI 完全給不了的體驗。「延誤自動重排」沒有對手做到。
- **為什麼吸引人**:旅途中一天打開八次的 app,回國後也不會被刪。
- **可行性**:資料都在,重排直接重用現有工具管線;需新增行動版視圖、離線快取、項目鎖定標記。反應速度是生死線。

### 9. 旅費分帳 ◆ 雙視角命中
每張行程卡可掛預估與實際花費,自動加總每日/總預算;旅途中一句話或拍收據記帳(「我付了四人晚餐 3200」),自動綁行程項目、按人拆分、換匯,結束時算出最少轉帳次數的結清方案。

- **為什麼吸引人**:多數人跳回 Google 試算表的最大理由就是「要算錢」;把錢綁在行程卡上,拆掉試算表最後一個存在理由。
- **AI 角色**:自然語言/收據照片 → 結構化帳目(降低輸入摩擦);拆分與結清演算法不需 AI。
- **可行性**:純 CRUD + 匯率 API,無技術風險;難點是記帳流程必須壓到三秒內完成,比開 Splitwise 更少步驟才有人用。

### 10. 訂單轉寄收件匣 ◆ 雙視角命中
每個行程有專屬 email(如 trip-abc@tripace.app),把訂房/機票/門票確認信轉寄過去,自動解析成時間軸項目,含訂位代號、地址、QR code。

- **取代的繁瑣流程**:出發前翻十個網站與信箱把代號地址複製到備忘錄,機場排隊時狂滑信箱找 QR code。
- **為什麼吸引人**:這是 TripIt 唯一的護城河,但 TripIt 只能排訂單不能規劃;「規劃 + 訂單」在同一條時間軸,使用者沒有理由開兩個 app。
- **AI 角色**:核心——各家確認信格式與語言都不同,LLM 解析非結構化 email 正是強項,規則式 parser 做不到。
- **可行性**:inbound mail(SES/Mailgun)成熟、LLM 抽取準確率高;難點在 PDF 附件與隱私。轉寄制比授權讀整個 Gmail 輕量。

## 整體觀察

- 十項合起來覆蓋完整旅程生命週期:靈感(1)→ 建立信任(2、3)→ 團體決策(6)→ 編排充實(4、5)→ 訂單與錢(9、10)→ 旅途中(8)→ 傳播獲客(7)。
- 其中 1、2、3、8 直接建立在 Tripace 已有的「LLM 工具操作時間軸」技術優勢上,是對手最難抄的;9、10 負責拆掉試算表與 TripIt 的存在理由;7 構成獲客迴圈。

## 附錄:未入選但值得記錄的項目

| 項目 | 一句話 | 未入選原因 |
|---|---|---|
| 追問式開局 | 開新行程時 AI 先像在地朋友追問 3~5 個關鍵問題,答案存成偏好卡 | 價值在「改善建議品質」,屬於體驗打磨而非獨立功能;可併入對話設計 |
| 旅行風格記憶 | 從歷次對話與實際修改累積長期偏好(總是刪美術館、走得慢) | 依賴使用量累積才有效,早期價值低;猜錯的記憶會污染建議 |
| 隨行翻譯卡 | 每個行程項目附當地語言卡(給司機看的地址、點餐句、菜單翻譯) | 價值高但與編排主線較遠,適合行動端成熟後再做 |
| 雨天 B 計畫 | 出發前監測天氣,對怕雨的日子預生成室內替代版 | 與「今天模式+現場救援」部分重疊,預案機制可作為其延伸 |
| 空檔交集喬日期 | 行程定案前讓成員圈可行日期,自動交集 | 純機制速贏,但發生在「行程存在之前」,與編排主線獨立 |
| 行前文件體檢 | 依國籍與目的地列簽證/入境卡要求與期限 | 錯誤代價高(過期資訊害人誤機),需人工校對維護,成本效益比低 |
| 看行程打包清單 | 讀行程生成勾選式行李清單,共用物品指派成員 | 好做但差異化低,獨立打包 app 眾多 |
| 行前倒數 Widget | iOS 桌面小工具顯示倒數與行前待辦 | 留存效益好但屬行銷層,依賴 iOS 端成熟 |
| 一頁行程海報 | 一鍵生成 IG 限動尺寸的行程海報,QR 連回完整行程 | 獲客價值高,但設計投入大於工程,適合產品定型後做 |
| 旅程回憶錄 | 旅程結束後相簿照片自動對回時間軸,生成圖文遊記 | 長期黏著關鍵,但依賴 iOS PhotoKit 與使用者已完成整趟旅程,屬後期功能 |
