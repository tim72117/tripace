# 行程編排操作介面與流程設計

設計 Web 前端(`web/src`)的行程編排體驗:對話式 AI 操作、手動編輯、推薦整合、
以及貫穿三者的**地理理解**呈現。本文是設計文件,不是實作;所有設計立足於
現有元件與後端能力(以 `ChatScreen.tsx`、`Timeline.tsx`、`wanttools/`、
`ROUTING_ARCHITECTURE.md` 盤點的現況為基準),凡需要新增後端能力或外部
API 之處,逐一標註成本與風險。

---

## 一、設計目標與原則

這份設計解決一件事:**讓使用者能又快又安心地「編排」行程,而不只是「記錄」行程**。
現況已能用一句話讓 AI 寫入條目、能在時間軸上看到結果,但三個斷層讓「編排」卡住:

1. 操作斷層——推薦景點看完就散(彈窗卡片沒有「加入行程」),手動編輯只有一張
   陽春表單,新增與刪除沒有手動入口;
2. 回饋斷層——AI 改了哪筆有光影動畫,但改完「對不對、要不要反悔」沒有著力點;
3. **地理斷層**——時間軸顯示「淺草寺 10:00 → 晴空塔 13:00」,使用者無從判斷
   這兩點在哪、多遠、順不順路,而這正是編排行程時最頻繁的疑問。

設計遵循的原則:

- **殊途同歸**:對話與手動兩種操作寫的是同一份 entry 資料、走同一套
  WS 事件、用同一套視覺回饋(`.updating` 光影、波浪佔位)。使用者不需要
  意識到「這筆是 AI 改的還是我改的」。
- **每個操作都有即時、可指認的回饋**:進行中看得到「哪一筆」正在變,
  完成後有回執,回執上有反悔(undo)的入口。
- **地理脈絡一瞥可得,但不常駐佔屏**:時間軸本身就內嵌最小限度的地理資訊
  (距離、移動時間),地圖是隨叫隨到的情境面板,而非永遠吃掉半個螢幕。
- **立足現況延伸**:沿用暖色調色盤(`--ios-bg #F5F2ED`、`--color-accent
  #C4956A`、`--color-dark #3D3530`)、既有時間軸三欄結構、既有彈窗/底部面板
  模式(`.rp-modal`、`.ask-user-sheet`、`.edit-entry-sheet`)、既有極簡地圖
  樣式(`RecommendedPlacesMap.tsx` 的 `MINIMAL_MAP_STYLE`)。不推翻重寫。
- **誠實面對資料與成本**:座標覆蓋率、Google API 計費、TOS 快取限制,
  都在文件內明說,不假設「反正之後會有」。
- **手機優先**:所有互動先以單手直式操作設計,桌面版是同一套機制的
  空間放大,不是另一套產品。

## 二、資訊架構與畫面佈局

### 2.1 三個資訊層的定位

| 層 | 角色 | 現況 | 本設計的定位 |
|---|---|---|---|
| **時間軸** | 主體、唯一權威畫面 | `MultiTrackTimeline`:日期欄+軸線+卡片,跨日條目(住宿等)為沙棕主線 | 維持主體;新增「移動間隙標籤」與「日摘要」兩種地理內嵌元素 |
| **對話** | 暫態指令通道 | 毛玻璃浮層(`chat-overlay`)+ 圓鈕 composer(推薦/對話) | 維持暫態浮層;新增「操作回執卡」承載確認與 undo |
| **地圖** | 情境面板 | 僅 debug demo(`RecommendedPlacesMap`),正式畫面無地圖 | 升級為三檔可收合面板,與時間軸雙向聯動 |

原則:時間軸永遠在,對話與地圖都是「需要時浮現、不需要時讓路」的第二層。
兩者互斥展開(手機上同時開對話浮層與地圖會互相遮擋,規則見 6.4)。

### 2.2 手機直式(< 768px)

預設狀態:在現有 navbar 與時間軸之間,插入一條高約 36px 的**地理脈絡列**
(map context bar)。它常駐、極輕,是地圖的收合形態,也是「不開地圖也知道
今天走多遠」的保底資訊。

```
┌───────────────────────────────┐
│ ‹  東京三天兩夜        ⋯ ⇪ 👥 │ ← navbar(現有,滾動自動隱藏)
├───────────────────────────────┤
│ ▸ 🗺 6/29 · 5 個地點 · 約 6.8 km │ ← 地理脈絡列(新):聚焦日摘要,點擊展開地圖

│ 29 ●─ 09:00 淺草寺            │
│    │   📍 台東區淺草2丁目      │
│    ┆  ↓ 1.4 km · 步行約 20 分  │ ← 移動間隙標籤(新,見 6.3)
│    ●─ 11:00 隅田公園          │
│    ┆  ↓ 2.3 km · 電車約 10 分  │
│    ●─ 13:00 晴空塔            │

地圖半開(點地理脈絡列、點卡片地點行、或點間隙標籤時):

│ ┌───────────────────────────┐ │

│   (✦ 推薦)   (💬 對話)         │
└───────────────────────────────┘

- 地圖面板從地理脈絡列原地向下展開(高度 transition,與 composer 展開輸入框
  同一種「原地生長」的動效語彙),不是蓋上來的 modal。
- navbar 的滾動隱藏行為不變;地圖半開時時間軸仍可滾動,聚焦日跟著換,
  地圖內容隨之切換(見 6.5)。
- 全螢幕檔(⤢)給「仔細看地圖」用:地圖佔滿 navbar 以下整個畫面,底部浮一條
  當日條目的橫向膠囊列表可左右滑,點膠囊聚焦 marker;左上「‹」退回半開。

### 2.3 桌面寬版(≥ 768px)

現況桌面版是「側欄 272px + 主內容(ChatScreen)」。分兩檔延伸:

- **768–1119px**:版面不變,地圖沿用手機的收合列/半開面板(同一套元件,
  不另做一份)。
- **≥ 1120px**:主內容右側分出常駐**地圖欄**(360–420px)。時間軸不再需要
  地理脈絡列的「展開」功能,該列退化為地圖欄頂部的日摘要。

┌────────┬──────────────────────────┬──────────────────┐
│ 行程    │ ‹ 東京三天兩夜     ⋯ ⇪ 👥 │   6/29 動線       │

│ ▪ 東京…│ 29 ●─ 09:00 淺草寺        │  │  ①───②     │  │
│ ▪ 京都…│    ┆ ↓1.4km·步行20分      │  │   \   \    │  │
│        │    ●─ 11:00 隅田公園      │  │   ③───④  ⌂ │  │
│        │    ●─ 13:00 晴空塔        │  │            │  │
│        │ 30 ●─ …                  │  └────────────┘  │

  272px        彈性(min 480px)          360–420px
```

- 地圖欄可由頂部按鈕收合(收合後主內容回到現在的置中版面),偏好記
  `localStorage`。
- 對話浮層(`chat-overlay`)維持蓋在時間軸欄上,不侵入地圖欄——AI 操作
  進行時,右側地圖同步亮起被操作的 marker,是桌面版獨有的「左右對照」體驗。

### 2.4 公開分享頁(/public/{token})

`PublicViewScreen` 沿用同一個時間軸元件,因此間隙標籤、地理脈絡列、地圖面板
(唯讀)可直接複用——訪客看別人分享的行程時,地理理解需求完全相同。限制:
公開頁沒有 WS(`handleWS` 走 `requireMember`),因此無即時更新,地圖為
純瀏覽;editable 公開頁送出後仍靠整頁 reload(現況行為)。

## 三、對話式 AI 操作流程

### 3.1 現況(基礎已具備)

一句話進 `assist` 後的既有回饋鏈:

1. 送出即插入海浪佔位泡泡(`WaveLoader`);
2. `task_plan` 建立多步計畫時,WS `task_created` 在對應日期插入逐字波浪
   佔位卡(`TaskPlaceholderCard`),`task_entry_ready` 換成正式卡;
3. `entry_update` 前 WS `entry_updating` 讓該卡亮「更新中」光影
   (最短 800ms),`entries_updated` 後重抓解除;
4. `ask_user`(目前僅日期)/`ask_choice` 由 WS 推事件開底部面板;
5. `recorded` 分支輪詢 `fetchEntries` 直到筆數變多(上限 20 秒)。

### 3.2 需要補齊的三段體驗

**(a) 進行中——讓「新增」與「刪除」也可指認。**
現況只有 update 有逐筆光影、只有走 task_plan 的新增有佔位卡。補齊:

- 單獨 `entry_add`(沒帶 taskID)完成時,前端以「重抓前後 entries 的差集」
  找出新卡,對它播放一次**進場動畫**(淡入 + 一輪光暈,約 1.2s),讓視線
  被帶到新增的位置;若新卡在視窗外,時間軸自動平滑捲動到它。零後端改動。
- `entry_delete` 完成時,同樣以差集找出消失的卡,在原位塌陷(高度收合
  200ms)並暫留一條 5 秒的「已刪除『○○』 [復原]」細條。零後端改動。
- 地圖已展開時,被操作條目的 marker 同步呼吸閃爍(同 `.updating` 的
  光影節奏),操作完成後 marker 定格——「AI 正在動哪個地點」在地圖上
  也看得到。

**(b) 完成後——操作回執卡(對話浮層內)。**
`assist` 回覆後,在對話浮層插入一張結構化回執(取代單純讓卡片自己出現):

```
┌─────────────────────────────┐
│ ✓ 已更新「晴空塔」            │
│    時間  13:00 → 15:00       │
│    [檢視]         [復原]     │
└─────────────────────────────┘
```

- 內容:動作(新增 N 筆/更新/刪除)+ 變更欄位的 before → after 摘要。
  update 的 before 值由前端在收到 `entry_updating` 時就地快照(該
  entryID 當下的欄位值),不需後端提供 diff。
- [檢視] 捲動到該卡並展開;[復原] 見 3.3。
- 回執與現有訊息泡泡同屬 `chat-overlay`,點時間軸空白處一併淡出(現有
  行為),但 undo 入口不因此消失——見下。

**(c) 反悔——undo 的分層設計。**
後端沒有版本歷史,undo 只能靠前端反向操作。誠實拆解可行性:

| 原操作 | 反向操作 | 現有端點夠嗎 |
|---|---|---|
| entry_update | PATCH 回 before 值 | ✅ `PATCH /v1/entries/{id}` 已有 |
| entry_add | 刪除該 entryID | ❌ 需新增 `DELETE /v1/entries/{id}`(小:復用 AI `entry_delete` 已在用的 `store.DeleteEntry`,外層包 requireEditor) |
| entry_delete | 以快照重建 | ❌ 需新增 `POST /v1/channels/{id}/entries`(小:復用 `tripsvc.Record`;同時就是手動新增的入口,見第四章) |

- 前端維護 per-channel 的 session undo stack(記憶體,重整即失,誠實標示
  「復原僅限本次瀏覽期間」);composer 上方浮一顆小小的「↩」膠囊,
  存在可復原操作時才顯示。
- 已知限制:`PATCH` 目前「空字串 = 不改」,若 before 值本來是空(例如
  原本沒有 end),undo 無法把欄位改回空。**完整的 undo 依賴 Phase 2 的
  API 語意修正**(建議:JSON 缺鍵 = 不改、`null` = 清空),在此之前
  回執卡對這類欄位標註「此欄位無法自動復原」。
- fallback:回執卡的復原失敗時,提示可改用對話(「跟 AI 說:把剛剛的
  變更改回來」)——agent 有對話歷史與 `entry_query`,通常能復原,但
  不保證精確,故只作為輔助路徑而非主設計。

### 3.3 AI 追問(ask_user / ask_choice)的呈現強化

現況兩個底部面板已可用,補三件事:

1. **關閉 ≠ 取消消失**:點背景關閉面板後,追問收合成 composer 上方的
   「待回覆 chip」(`📌 待補:希爾頓退房日`),點 chip 重開面板;回覆
   送出後 chip 消失。解決「誤觸背景、追問就再也找不回」——現況 agent
   已結束該輪,資訊永遠缺著。
2. **帶地理脈絡的選擇**:`ask_choice` 的選項若對應地點(例如「兩間旅館
   選一間」),面板內嵌一張迷你地圖同時標出選項位置與當日行程動線,
   讓「選哪個順路」變成看得見的判斷(選項需帶 lat/lng,需後端在
   `ask_choice` options 增列選填座標欄位,小改)。
3. `askType` 目前僅 `date`;隨手動編輯元件的建立,`time`、`place` 兩型
   可直接復用同一批輸入元件(Phase 2)。

### 3.4 時序總覽(一句話 → 時間軸更新)

```
使用者送出「把晴空塔改到三點」
  → 泡泡佔位(海浪)                              [即時]
  → WS entry_updating(ent_xxx)                  [~1s]
      → 卡片光影 + 地圖 marker 呼吸 + 快照 before
  → WS entries_updated → 重抓 → 光影解除          [~2-5s]
  → assist HTTP 回覆 → 回執卡(13:00→15:00 [復原])
  → 間隙標籤與日摘要隨新時間重算                   [即時,前端]
```

改良建議(零後端):`recorded` 分支目前用 1 秒輪詢等新條目,但後端本來
就會廣播 `entries_updated`——前端改成「以 WS 事件為主、輪詢僅作 20 秒
保底」,平均等待可縮短一半以上。

## 四、手動編輯流程

### 4.1 編輯一筆(現有 EditEntrySheet 的延伸)

入口不變:點卡片展開 → 「編輯」→ 底部表單(`edit-entry-sheet`)。表單升級:

```
┌───────────────────────────────┐
│ 編輯條目                   ✕  │
│ 標題 [晴空塔                ] │
│ 開始 [2026-06-29] [15:00  ▾] │ ← 時刻欄附快速 chips:上午9:00/午12:00/下午14:00/晚18:00
│ 結束 [          ] [       ]  │
│ 類型 [活動 ▾]                 │ ← kind(flight/stay/car/activity/food/transport/other)
│ 地點 [東京晴空塔            ] │
│  ┌─────────────────────────┐ │
│  │   📍(120px 靜態小地圖)    │ │ ← 地點有座標時顯示;改地點後顯示「儲存後將重新定位」
│  └─────────────────────────┘ │
│ 備註 [                     ] │
│ [🗑 刪除]        [取消][儲存] │
└───────────────────────────────┘
```

- **地點欄與座標**:誠實現況——`PATCH` 改了 location 後座標不會更新
  (`handleUpdateEntry` 以 `tripsvc.New(s.store, nil)` 建 service,geo 為
  nil)。需後端小改:PATCH 帶 location 且值有變時,走與 `tripsvc.Record`
  相同的非同步 geocode 補 lat/lng。這是地圖正確性的必要條件,列 Phase 1。
- **同步廣播**:誠實現況——`handleUpdateEntry` 完成後**沒有**廣播
  `entries_updated`,其他成員(與自己的第二個分頁)不會即時看到手動修改。
  後端補一行 `NotifyEntriesUpdated`,列 Phase 1。
- **刪除鈕**:需 `DELETE /v1/entries/{id}`(同 3.2 undo 所需,一魚兩吃)。
  按下後卡片走與 AI 刪除相同的塌陷動畫 + 5 秒復原條。
- 欄位清空限制(空字串=不改)如實揭露在 UI:留空的欄位顯示 placeholder
  「留空表示不改」(現況已有),Phase 2 隨 API 語意修正後開放清空。

### 4.2 新增一筆

兩個入口,對應兩種心智:

- **從日期新增**:時間軸每個日期標籤(`tl-date-day`)可點,彈出該日操作
  選單:「＋ 新增一筆」「🗺 看這天的地圖」。新增開啟空白版編輯表單,
  日期預填該日。空白日(blank row)整列可點,行為相同——把「這天還沒
  安排」變成入口而非死區。
- **從推薦新增**:見第五章。

寫入走 Phase 2 的 `POST /v1/channels/{id}/entries`;在端點存在之前
(Phase 1),表單「儲存」改為組一句結構化訊息送 `assist`(「6/30 15:00
新增『teamLab』,地點:豐洲」),由 AI 寫入——體驗一致,只是稍慢,
表單按鈕顯示海浪 loading。這讓新增 UI 可以先行,不被後端排程卡住。

### 4.3 調整時間與順序

時間軸的排序完全由 `start + startTime` 決定(無獨立 order 欄位),因此
「調順序」本質是「改時間」。設計三個層次,由輕到重:

1. **快速時間 chips**(Phase 1):卡片展開後,編輯鈕旁併排「⏰ 改時間」
   直接彈 time picker + 常用時段 chips,不必進完整表單。改完即走 PATCH
   + `.updating` 光影。
2. **跨日拖曳**(Phase 2):長按卡片 300ms 進入拖曳,拖到另一個日期區段
   放開 = 改 start 日期、時刻不變。語意清楚、無歧義,是拖曳的主場景。
   拖曳中目標日期列高亮,放開後卡片在新位置播進場動畫。
3. **同日拖曳排序**(Phase 2,謹慎):放到同日兩卡之間時,新時刻取前後
   兩卡時刻的中點(10:00 與 14:00 之間 → 12:00);前後同刻或無法取中時,
   放開後立即彈 time picker 請使用者定奪。誠實標註:這條手勢的邊界情況多
   (全日事件無時刻、跨日主線條目不參與拖曳),實作成本高於價值的疑慮
   存在,若 Phase 2 資源緊,同日排序可先交給對話(「把晴空塔排到淺草寺
   前面」)——AI 改時間本來就是這套系統的強項。

### 4.4 手動與 AI 的視覺語言一致

- 手動儲存時,前端本地把該 entryID 加進 `updatingEntryIDs`(與 WS
  `entry_updating` 同一個 state),完成後解除——光影動畫與 AI 路徑
  完全同款。
- 手動操作同樣產生回執卡與 undo 記錄(3.2 的 stack 不分來源)。
- 差異僅在觸發端:沒有海浪泡泡(因為沒有對話),表單/按鈕自身的
  loading 態承擔「進行中」。

## 五、推薦整合流程

### 5.1 現況鏈路與缺口

`✦ 推薦`鈕 → 固定語句「推薦附近的景點」進 assist → agent 呼叫
`recommend_nearby`(Places Nearby,回 name/address/lat/lng/primaryType)
→ WS `recommended_places` → `RecommendedPlacesModal` 卡片列表。

缺口:(1) 卡片純展示,**沒有任何動作**,看完關掉就散;(2) 有座標卻沒
地圖,「這些推薦在我行程的哪一邊」無從判斷;(3) 推薦中心點由 agent 從
語句猜,沒有利用畫面上的聚焦日脈絡。

### 5.2 設計:推薦面板 = 地圖 + 卡片 + 加入

`RecommendedPlacesModal` 升級為上圖下卡的雙區面板(手機底部滿寬彈起,
桌面沿用置中 modal;地圖複用 `RecommendedPlacesMap`,終於從 demo 轉正):

```
┌───────────────────────────────┐
│ 推薦景點 · 8 個(晴空塔周邊)  ✕ │
│ ┌───────────────────────────┐ │
│ │    ◇a   ①───②             │ │  ● 數字=當日既有行程(沙棕)
│ │       ◇b  \                │ │  ◇ 字母=推薦候選(琥珀)
│ │    ◇c      ③───④      ⌂  │ │  既有動線同圖呈現 → 順路一眼可辨
│ └───────────────────────────┘ │
│ ◀ ┌─────────┐ ┌─────────┐ ▶  │
│   │ [照片]   │ │ [照片]   │    │  卡片橫向滑動(手機)/兩欄(桌面)
│   │ a 仲見世 │ │ b 今戶神社│    │  滑到哪張,地圖對應 ◇ 放大
│   │ ★4.4    │ │ ★4.3    │    │
│   │ 距② 400m │ │ 距② 1.1km│    │ ← 與最近行程點的直線距離(前端算)
│   │ [＋ 加入] │ │ [＋ 加入] │    │
│   └─────────┘ └─────────┘    │
└───────────────────────────────┘
```

- **與行程對照**是本面板的靈魂:把聚焦日既有條目(灰調編號 marker +
  細動線)畫在同一張地圖上,推薦候選用琥珀菱形區隔。「b 就在 ② 旁邊、
  a 要過河」不需要任何文字說明。
- 卡片增列「距最近行程點 ○○」——用既有條目座標與候選座標做 Haversine,
  純前端,零成本。
- 推薦鈕的固定語句升級為帶脈絡:「推薦 {聚焦日已有條目的地區} 附近的
  景點」;若使用者先選中了某張卡片再按推薦,則以該卡地點為中心
  (「推薦晴空塔附近的餐廳」式的語句由前端組出,agent 端零改動)。

### 5.3 「加入」之後:時間怎麼排

點「＋ 加入」:

1. 彈極簡確認條(非整頁表單):`加入哪一天? [6/29(聚焦日)▾]  [加入並讓
   AI 排時間] [自己選時間]`。
2. **AI 排時間**(預設、推薦):前端組語句送 assist——「把『今戶神社』
   (東京都台東區今戶1-5-22,lat/lng)加入 6/29 行程,幫我排合理時間」。
   agent 的既有時間推斷規則(景點上午 9:00/下午 14:00 等,見
   `assistant_agent.go`)會排出時刻,還會在與既有條目衝突時追問。面板
   不關閉,該卡片變成「加入中…」海浪態,WS `entries_updated` 後打勾,
   時間軸同步出現新卡進場動畫。可連續加入多個。
3. **自己選時間**:開 4.2 的新增表單,標題/地點/座標預填。
4. 誠實標註:Phase 1 靠 assist 寫入,座標能否落地取決於 6.2 的座標補齊
   方案(`entry_add` 目前不收 location/座標,是必須先修的洞);修好後,
   加入語句中的座標可由 agent 原樣寫入,不必重查 geocode。

## 六、地理理解設計(重點)

### 6.1 三層地理問題,三種呈現強度

| 問題 | 使用者情境 | 呈現 | 何處 |
|---|---|---|---|
| 這個點在哪? | 看到「淺草寺」想確認位置 | 卡片地點行、地圖單點聚焦、外開導航(現有 NavButton) | 時間軸卡片 + 地圖 |
| 這兩點多遠? | 「淺草寺 10:00 → 晴空塔 13:00」中間夠不夠 | **移動間隙標籤**(距離+粗估時間) | 時間軸內嵌 |
| 這一天順不順? | 排完一天想檢查動線 | 日摘要 + 地圖動線 + 繞路提示 | 地理脈絡列 + 地圖 |

分層的用意:第一、二層**不開地圖就能得到答案**(內嵌在時間軸),地圖
只為第三層與「想仔細看」的時刻服務——這是「地理脈絡一瞥可得,但地圖
不常駐佔屏」原則的具體化。

### 6.2 資料基礎:座標覆蓋現況與補齊(一切的前提)

誠實盤點三條寫入路徑的座標行為:

| 寫入路徑 | location 落地? | lat/lng 落地? |
|---|---|---|
| AI 對話(`entry_add` → `BindSink` → `InsertEntry`) | ❌ `RecordedEntry` 根本沒有 Location 欄位 | ❌ agent 雖被指示先 `geocode` 確認座標,但查到的座標**只呈現、不儲存** |
| CLI/internal(`tripsvc.Record`) | ✅ | ✅ 非同步 geocode 補寫 |
| 手動 PATCH(`handleUpdateEntry`) | ✅ | ❌ geo 注入為 nil,改地點不重新定位 |

**結論:目前主力路徑(AI 對話)寫入的條目沒有座標,地圖什麼都畫不出來。**
這不是 UI 問題,是 Phase 1 必須先修的資料洞:

1. `entry_add` 工具宣告增加 `location`、`lat`、`lng` 參數(agent 流程本來
   就先跑 geocode,順手帶入即可,prompt 小改);`RecordedEntry` 與
   `BindSink` 對應補欄位。
2. `BindSink` 的寫入改走(或比照)`tripsvc.Record`:有 location 而無座標時
   非同步 geocode 補寫,完成後廣播 `entries_updated` 讓前端刷新出 marker。
3. `handleUpdateEntry` 注入 geo client,location 變更時重新定位(4.1 已述)。
4. 存量條目:提供一次性補跑腳本(遍歷有 location 無 lat/lng 的 entry 呼叫
   geocode);量小,一次付費在免費額度內。
5. 前端對「無座標條目」的優雅降級:卡片照常,地圖面板該點缺席並在日摘要
   註記「1 個地點未定位 [補定位]」,點擊觸發 geocode(經後端代理)。

### 6.3 時間軸內嵌:移動間隙標籤與日摘要

**移動間隙標籤**(transit gap)——插在同日相鄰兩張(皆有座標的)卡片之間,
掛在軸線上,視覺上是虛線小節點 + 一行灰字,不佔卡片位、不搶視覺:

```
●─ 10:00 淺草寺
┆   ↓ 1.4 km · 步行約 20 分          ← 預設:直線距離 × 1.3 的粗估
●─ 13:00 晴空塔
```

- 資料:兩點 Haversine 直線距離(純前端、零成本);粗估時間 = 修正距離
  (×1.3 路網係數)÷ 速度(步行 4.5km/h;>2km 顯示「車程約 X 分」以
  30km/h 估)。文案永遠帶「約」,tooltip 註明「依直線距離粗估」——
  不假裝精確。
- **時間可行性檢查**:前一項有結束時間(或依 kind 推得停留時間)時,
  若「前項結束 + 粗估移動」晚於後項開始,標籤轉磚紅:
  `⚠ 移動約 40 分,but 間隔只有 20 分`,點擊展開兩個動作:
  「讓 AI 調整時間」(組語句送 assist)/「自己改」(開 time picker)。
  粗估有誤差,警示閾值取保守(超出 1.5 倍才亮紅),避免狼來了。
- 點標籤本身 → 地圖半開並聚焦這兩點(fitBounds 到兩 marker + 連線加粗)。
- 精確模式(Phase 2 接 Routes API 後):標籤升級為實際路徑的
  `🚶 18 分(1.6 km)`或`🚇 12 分`,資料來源與成本見 6.7。

**日摘要**——地理脈絡列(手機)/地圖欄頂部(桌面)常駐:
`6/29 · 5 個地點 · 約 6.8 km`;Phase 3 加繞路提示(見 6.6)。

### 6.4 地圖面板的狀態機:什麼時候出現、什麼時候讓路

三檔:`collapsed`(36px 脈絡列)/ `half`(40dvh)/ `full`(全螢幕)。

自動展開到 half 的時機(皆為使用者的地理意圖訊號):

- 點地理脈絡列、卡片的地點行(📍)、移動間隙標籤;
- 推薦面板開啟時(面板自帶地圖,主地圖不動;關閉推薦後若使用者在推薦裡
  點過地圖,主地圖保持 half 以延續脈絡);
- 點 AI 回執卡的 [檢視] 且該條目有座標。

自動收合回 collapsed 的時機(讓路原則):

- 開啟任何底部面板(編輯表單、ask_user/ask_choice)或 composer 輸入框
  取得焦點(鍵盤要上來了);
- 對話浮層展開且訊息高度超過半屏(對話與地圖互斥,對話優先——使用者
  正在下指令);
- 手動:點面板頂部的收合把手、或在地圖上往上滑(手勢與底部面板一致)。

**不**自動收合的時機:時間軸滾動換聚焦日(只換內容不收面板——使用者
可能正在「邊滾邊看每天動線」,這是核心瀏覽模式);AI 操作進行中(要看
marker 回饋)。

記憶:檔位不持久化(每次進頻道從 collapsed 開始),桌面 ≥1120 的地圖欄
開關才記 `localStorage`——手機上「上次開著就一直開著」容易變成永久佔屏,
違背原則。

### 6.5 時間軸 ↔ 地圖聯動細節

**聚焦日(focus day)**:時間軸滾動時,取視窗垂直中線相交的
`tl-grid-row` 的日期,debounce 150ms。聚焦日決定:地理脈絡列摘要、
地圖顯示的動線、推薦的預設脈絡。點日期標籤可鎖定(再點解鎖回到滾動
跟隨),鎖定時脈絡列顯示 📌。

**卡片 → 地圖**:點卡片地點行 → 面板展開(若收合)→ 地圖 pan/zoom 至該
marker(zoom 上限 16,避免貼到屋頂)→ marker 放大 1.25x、深棕描邊,
其餘 marker 降飽和。再點一次或點地圖空白處還原全日 fitBounds。

**地圖 → 卡片**:點 marker → 時間軸平滑捲動至該卡並自動展開細節;
full 檔則是底部膠囊列表滑到對應項。點推薦候選 ◇ → 卡片區滑到對應卡。

**marker 視覺**(沿用調色盤,`AdvancedMarkerElement` 自訂 DOM):

- 一般條目:圓形、沙棕 `#C4956A` 底白字,**依當日時序編號 1..n**——
  編號是「順序連線」之外第二條理解動線的線索;
- 住宿(kind=stay):⌂ 形、森綠 `#5A8A6A`——它是每天的起點與終點錨,
  跨日常駐(當日動線的第 0 點與第 n+1 點);
- 推薦候選:菱形、琥珀 `#D9A441`(調色盤延伸色,與磚紅錯誤色區隔);
- 選中態:放大 + 深棕 `#3D3530` 描邊;更新中:呼吸光影(同 `.updating`)。

**動線(polyline)**:當日條目按時序以 2px 沙棕實線相連(含宿→首點、
末點→宿);相鄰兩天的「昨日末點→今日首點」在需要時(全螢幕檔的跨日
模式)以淺灰虛線畫出。誠實:這是**直線示意**,不是路網路徑——直線已
足以回答「順不順」,實際路徑留給 Phase 2 的 Routes API(且只在使用者
點某段 gap 時才取,不整天預載)。

**與 AI 的聯動**:`entry_updating` → 對應 marker 呼吸;`entries_updated`
→ 重畫動線(新卡 marker 彈跳一次進場)。在桌面三欄下,「左邊對話、
中間卡片光影、右邊 marker 跳動」構成一體的操作回饋。

### 6.6 一日動線與順路視覺化

- **看出繞路(被動)**:編號 marker + 順序連線本身就是順路檢查器——
  連線交叉、來回折返,一眼可見。這是 Phase 1 就有的能力,零額外資料。
- **點出繞路(主動,Phase 3)**:前端以直線距離計算當日訪問順序總長
  d_actual 與最短排序 d_best(n ≤ 8 全排列,n > 8 最近鄰 + 2-opt;
  端點固定為住宿錨點)。`d_actual / d_best > 1.4` 時,日摘要亮提示:
  `⚠ 動線可省約 2.1 km [讓 AI 重排]`。點擊組語句送 assist(帶建議
  順序與各條目 entryID),agent 以 `entry_update` 批次改 startTime,
  卡片逐筆光影、地圖動線漸變成新順序。
  誠實標註:直線最佳 ≠ 路網最佳,更 ≠ 考慮營業時間/預約的最佳;
  故僅在「明顯」繞路(閾值 1.4,保守)時提示,文案用「可省約」不用
  「最佳」;重排永遠經使用者確認(AI 回執 + undo 兜底),不自動執行。
- **跨日均衡(Phase 3,選配)**:全螢幕地圖的「整趟模式」以三色淡染
  各日活動範圍 convex hull,一眼看出「第 2 天跑了兩個區」;僅視覺,
  不做自動搬移建議,避免過度工程。

### 6.7 距離與移動時間的資料來源(誠實評估)

| 方案 | 品質 | 成本 | 取捨 |
|---|---|---|---|
| **Haversine 直線 × 1.3**(前端) | 距離粗估可用;時間誤差大(市區 ±50%) | $0 | Phase 1 全面採用;文案帶「約」與 tooltip 揭露 |
| **Google Routes API**(computeRoutes / computeRouteMatrix) | 實際路網 + 交通方式(步行/駕車/**大眾運輸**——日本旅遊場景電車時間是剛需) | 2025 改版後 Essentials 級約 **$5 / 1000 次**(元素),每 SKU 每月約 10,000 次免費;Transit 屬 Advanced 級,單價更高(約 $10/1000)。以官方定價頁為準 | Phase 2:僅在使用者展開 gap 詳情或進入精確模式時呼叫,經後端代理 + 快取 |
| OpenRouteService(免費方案) | 步行/駕車尚可,**無日本大眾運輸** | 免費 quota(~2000 次/日) | 可作步行/駕車的零成本升級,但解不了電車時間 |
| 自架 OSRM/Valhalla | 步行/駕車好;大眾運輸需另接 GTFS,工程大 | 主機與維運成本 | 量大之後才值得,現階段不建議 |

**量級推演**(說明為何「快取」是成敗關鍵):一趟 10 天行程、每天 6 點
→ 相鄰對 5 × 10 = 50 對;首次瀏覽 50 次元素呼叫,**快取命中後為 0**。
月活 100 個行程 × 平均重算 3 次(編輯改時間不改地點不需重算;改地點/
增刪點才失效對應段)≈ 15,000 元素/月——略超免費額度,支出在 $25/月
量級。反之若不快取、隨滾動重算,單一使用者一天就能刷掉上千次。

**快取設計與 TOS 風險**:後端新端點 `POST /v1/geo/route-summary`
(批次收 [origin,destination,mode] 對,回 distance/duration),DB 快取
鍵 = 座標各四捨五入到 ~11m(4 位小數)+ mode。誠實標註:Google TOS 對
Routes 結果的儲存有限制(多數內容僅允許暫時性快取;經緯度 geocode 結果
允許存至 30 天)。**建議只持久化 distance/duration 兩個純量、TTL 30 天,
不存 polyline**;上線前過一次 TOS 覆核,若認定不可行,退回 session 內
記憶體快取(成本估算相應上修約 3 倍)。前端另設節流:同一 gap 的精確
查詢結果掛在 entry pair 上,面板重開不重查。

**分級策略總結**:所有 gap 預設直線粗估(免費、即時)→ 使用者點開某段
gap 或開啟「精確模式」開關時,該段升級為 Routes 實測(步行 + 大眾運輸
兩行並列)→ 已升級的段落在座標不變時永久沿用快取。成本花在使用者
明確想要精確的那幾段,而不是每一段。

## 七、完整操作情境 walkthrough:「規劃東京三天兩夜」

**0. 建立行程。** 桌面側欄(或手機列表)「＋」建「東京三天兩夜」。
進入後時間軸空白:「在下方輸入記事,會依時間排列在這裡。」地理脈絡列
未出現(無任何座標條目時隱藏,不佔位)。

**1. 一句話立骨架。** 點 💬(圓鈕原地展開輸入框),輸入「6/29 到 7/1
去東京,住淺草的旅館,第一天下午到」。海浪泡泡 → agent 走 stay 流程,
缺退房日以外的資訊皆足,`ask_user` 不觸發;`task_plan` 佔位卡「住淺草
旅館(新增中)」波浪浮現在 6/29 → 正式卡落地:主線(沙棕跨日線)從

## 八、狀態與邊界情況

| 情境 | 現況 | 設計 |
|---|---|---|
| **初次載入** | 空白後直接出現 | 時間軸骨架屏(灰條日期 + 灰卡 × 3,300ms 內完成則跳過);地理脈絡列待 entries 到位才淡入 |
| **AI 推論中** | 海浪泡泡 + 佔位卡;recorded 靠 1s 輪詢(上限 20s) | 保留;改「WS `entries_updated` 為主、輪詢保底」縮短等待(3.4);逾時文案由沉默改為「AI 仍在處理,稍後條目會自動出現」+ 回執補送 |
| **WS 斷線** | `onclose` 無重連,斷了就永久失聯(僅 cleanup close) | 指數退避重連(1s→2s→…→30s 上限);navbar 下細黃條「連線中斷,顯示最後同步內容(HH:MM)」;重連成功 → `fetchEntries` 補償 + 黃條退場。斷線期間 HTTP 仍可用:手動編輯照常,AI 操作可送出但進行中回饋缺失(佔位卡/光影不出現),回執卡照常由 HTTP 回覆產生——降級但不阻斷 |
| **離線**(`navigator.onLine` = false) | 各請求各自報錯 | 唯讀模式:時間軸/地圖(已載入圖磚)可瀏覽,composer 與編輯入口禁用 + 「離線,僅供瀏覽」;恢復後自動重抓。不做離線寫入佇列(多人資料,離線排隊衝突風險 > 價值) |
| **多人同時編輯** | last-write-wins;PATCH 只覆蓋有帶的欄位(天然縮小衝突面);手動 PATCH 甚至不廣播 | Phase 1:PATCH 補廣播後,他人變更即時反映;自己的編輯表單開啟期間收到該 entry 的變更 → 表單頂黃條「此條目剛被其他成員更新 [載入新值/繼續(將覆蓋)]」。Phase 3:entry 加 `updatedAt`(現只有 createdAt,需 migration),PATCH 帶 baseUpdatedAt → 不符回 409 → 前端逐欄對比對話框 |
| **地圖不可用**(無 VITE_GOOGLE_MAPS_API_KEY / 載入失敗) | rp-map-error 文案 | 地理脈絡列與間隙標籤照常(Haversine 不依賴 Maps SDK);點開地圖位置顯示錯誤卡與重試;功能性退化不連坐 |

| **ask_user 被關閉** | 追問永久消失 | 待回覆 chip(3.3) |
| **公開頁** | 無 WS、reload 更新 | 唯讀地圖與間隙標籤可用;不做即時,頁面標註「內容為載入當下版本」 |

## 九、分階段落地建議

### Phase 1:「看得懂」——地理脈絡與回饋補全

前端:地理脈絡列 + 地圖面板(collapsed/half/full)+ 聚焦日聯動 + 編號
marker/動線(直線)+ 間隙標籤(Haversine 粗估)+ 時間可行性警示 +
推薦面板併入地圖與「加入(走 assist)」+ 回執卡與 update-undo +
新增/刪除的差集動畫 + WS 重連 + 待回覆 chip + 快速時間 chips。

後端依賴(皆為小改,但缺一不可):

1. `entry_add` 收 location/lat/lng 並落地;BindSink 路徑補非同步 geocode
   (**座標覆蓋,一切地圖功能的前提**);
2. `handleUpdateEntry` 補廣播 `entries_updated` + location 變更時重新 geocode;
3. 存量條目座標補跑腳本。

風險:Maps JS API 動態載入量(約 $7/1000 次,月免 10k)——地圖 lazy
mount(首次展開才載)+ 單例復用,一般使用量在免費額度內。

### Phase 2:「改得動」——手動編輯閉環與精確資料

前端:新增/刪除表單直連 REST、完整 undo(含 add/delete)、跨日拖曳
(+ 視資源做同日拖曳)、gap 精確模式(步行/大眾運輸並列)、
編輯表單小地圖、ask_user 擴充 time/place。

後端依賴:

1. `POST /v1/channels/{id}/entries`、`DELETE /v1/entries/{id}`
   (復用 tripsvc/store 既有函式,handler 級新增);
2. PATCH 語意修正:缺鍵=不改、null=清空(undo 完整性所繫);
3. `POST /v1/geo/route-summary` 代理 + DB 快取(**Routes API 計費與
   TOS 快取限制見 6.7,上線前過一次覆核**);
4. assist 完成事件改由 WS 全量推送,前端徹底移除輪詢。

### Phase 3:「排得好」——主動的地理智慧與多人強健性

前端:繞路偵測與 AI 重排、跨日均衡視覺、ask_choice 迷你地圖、
409 衝突對比 UI。

後端依賴:entry 加 `updatedAt` + 樂觀鎖(migration);`ask_choice`
options 帶座標;(選配)重排建議的伺服器端計算。

風險:繞路提示的信任成本——直線距離的建議若常「不準」會反噬信任,
故閾值保守、文案克制、永遠可復原,先小流量驗證再放大。
